## Goal

Port the Next.js auth flow + customer account page + full admin dashboard from the v0 repo into this TanStack Start project, backed by Lovable Cloud (Supabase under the hood) instead of the original raw Supabase setup.

## Phase 1 ‚Äî Enable Lovable Cloud + schema

Enable Lovable Cloud, then create:

- `profiles` table ‚Äî `id` (uuid PK = auth user id), `name`, `email`, `created_at`. Auto-created on signup via trigger. RLS: user reads/updates own row; admins read all.
- `app_role` enum: `admin`, `customer`.
- `user_roles` table ‚Äî `(user_id, role)`. RLS-enforced, plus `has_role(uuid, app_role)` SECURITY DEFINER function (security best practice ‚Äî no roles on profiles).
- `products` table ‚Äî id, name, name_ar, category, price, original_price, image, description, description_ar, stock, in_stock, is_new, discount, rating, reviews. Seed with the 8 existing products from `src/lib/products.ts`. RLS: public read; admin write.
- `orders` table ‚Äî id, user_id, items (jsonb), total_amount, status (`pending|processing|shipped|delivered|cancelled`), shipping_address (jsonb), created_at. RLS: user sees own orders; admin sees/updates all.

The original repo's `admin_users` table is replaced with the safer `user_roles` + `has_role()` pattern.

## Phase 2 ‚Äî Auth foundation

- `src/integrations/supabase/client` is auto-generated by Cloud ‚Äî use it directly (no service-role needed for these flows).
- Rewrite `src/lib/auth-context.tsx` with `useState`/`useEffect`, `onAuthStateChange` listener set up BEFORE `getSession`, exposing `user`, `session`, `loading`, `isAdmin`, `login`, `register`, `logout`.
- Wrap `<CartProvider>` inside an `<AuthProvider>` in `src/routes/__root.tsx`.

## Phase 3 ‚Äî Customer auth + account routes

New route files (each with `head()` SEO metadata):

- `src/routes/login.tsx` ‚Äî email/password login form, redirects to `/account` on success. Glass card styling matching original.
- `src/routes/register.tsx` ‚Äî name/email/password signup with `emailRedirectTo: window.location.origin`.
- `src/routes/account.tsx` ‚Äî small customer dashboard: profile card (name, email, member since, logout) + order history list pulled from `orders` table filtered by `user_id`. Shows order id, date, status badge, items, total. If not signed in ‚Üí redirect to `/login`.
- Update `src/routes/checkout.tsx` to insert into `orders` table when user is signed in (instead of localStorage).
- Update `src/components/site/header.tsx` to show Login/Register links when signed-out, Account/Logout when signed-in, and an "Admin" link when `isAdmin`.

## Phase 4 ‚Äî Admin dashboard

Layout route `src/routes/admin.tsx` (TanStack pathless-style guard):
- `beforeLoad` checks session + `has_role(user, 'admin')`; redirects to `/admin/login` if not admin.
- Renders shared admin shell: header with logo, nav links (Dashboard / Orders / Products / Users), logout, `<Outlet />`.

Routes:

- `src/routes/admin.login.tsx` ‚Äî separate styled admin login (glass card, gradient blobs from original). Verifies admin role after sign-in; signs out + shows error if not admin.
- `src/routes/admin.index.tsx` (or `admin.dashboard.tsx`) ‚Äî KPI cards: total orders, total revenue, total products, total users. Quick links to sub-pages.
- `src/routes/admin.orders.tsx` ‚Äî list all orders, change status via dropdown, delete order.
- `src/routes/admin.products.tsx` ‚Äî full CRUD: list, add (dialog), edit (dialog), delete with confirm. Fields match products schema.
- `src/routes/admin.users.tsx` ‚Äî list profiles + their role; allow toggling admin role (insert/delete row in `user_roles`).

All admin data ops go directly through the Supabase client; RLS + `has_role()` enforce server-side authorization.

## Technical notes

- Original uses Next.js `@supabase/ssr` with cookie-based SSR ‚Äî replaced with TanStack Start client-side Supabase + `_authenticated` style guards via `beforeLoad`.
- Roles live in `user_roles` (NEVER on profiles) to prevent privilege escalation, per security guideline.
- No `service_role` key in client code anywhere.
- The original's `setup-demo` / hardcoded `admin@nour.tn` credentials are dropped ‚Äî first admin will be promoted manually via SQL after signup (I'll provide the one-liner).
